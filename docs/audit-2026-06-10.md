## 3. Issue #22 (idle freeze) — root-cause analysis

Symptom: soft hang while idle on a heavy-vault-churn day (5 sessions
writing logs); no `.ips` report. Verified mechanism candidates, ranked:

### 3.1 Undebounced full-file re-parse per FSEvent on the main actor (primary)

- `SessionLogService.startWatching()` calls `reconcile()` on **every**
  `.log` event, no debounce (`SessionLogService.swift:427-429`).
- `reconcile` → `parseBody`: synchronous `Data(contentsOf:)` of the whole
  log + ~5 fresh `NSRegularExpression` compilations + full error scan
  (`SessionLogService.swift:138-213, 262-278`) — on the main actor.
- Scout session logs are appended continuously while a run executes.
  FSEvents latency is 0.1s and `FileWatcher`'s `AsyncStream` buffers
  unboundedly (`FileWatcher.swift:39-49`), so under sustained appends the
  main actor can fall permanently behind → beachball, no crash report.
  Matches the reported symptom precisely.

### 3.2 Four independent watchers on `.scout-logs/` (amplifier)

SessionLogService, ConnectorHealthService (`:249`), UsageTrackerService
(`:39-44`), SessionTokensService each watch the same directory and
re-read files per event. Multiplies per-event cost; each FSEventStream
gets its own dispatch queue (`FileWatcher.swift:58`).

**Counter-example done right:** `ActionItemsDocumentService.swift:77-88`
— 250ms debounce + filename filter. This is the template.

### 3.3 SystemProcessRunner pipe wedge (latent bug, not the freeze)

`GitService.swift:222-229` reads stdout only inside `terminationHandler`.
A child writing > ~64KB before exit blocks on write → never terminates →
continuation never resumes → silently wedges the awaiting task
(schedule refresh; git commits in ActionItemsWriter's serialized queue).
Won't beachball the UI (callers await), but is a time bomb as scoutctl
JSON grows. Fix: drain pipes concurrently (`readabilityHandler` /
async `bytes`).

### Fix plan

1. Debounce + coalesce per file path in `SessionLogService` (mirror
   ActionItems pattern).
2. Move `parseBody` off the main actor (already a static pure function);
   hop back only to publish `runs`.
3. Cache compiled regexes as statics (also in `ActionItemsParser`, which
   compiles 10+ per parse, `ActionItemsParser.swift:197-231`).
4. Optional: parse only the log tail for the finish-marker scan.
5. Fix the pipe-drain pattern in `SystemProcessRunner`.
6. Land with the missing `reconcile()` end-to-end test (BACKLOG already
   flags it) — needs a controllable `FileSystemEventSource` test double.
7. If the freeze recurs pre-fix: capture `sample Scout` / spindump to
   confirm the main-thread stack.

### Strategic flag — one embedding decision, not three

\#18 Stage 2 (embedded Agent-SDK assistant), F-1 (embedded SwiftTerm
terminal), and BACKLOG's "view/interact with a running scout session"
(research-gated) are three framings of the same architectural question:
*how does a live Claude session run inside or attach to Scout.app?*
Write one shared spec/brainstorm before building any of the three, or
the app ends up with two embedding mechanisms.
